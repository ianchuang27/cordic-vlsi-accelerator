# Fixed-Point Pipelined CORDIC Accelerator
 
## Project Overview

For this VLSI final project, I implemented a fixed-point CORDIC accelerator for sine and cosine computation. The design uses CORDIC rotation mode, so the hardware computes the result with shifts, additions/subtractions, pipeline registers, and a small arctangent lookup table instead of multipliers.

I chose CORDIC because it fits the kind of DSP hardware we have been studying: it uses a simple shift-add datapath, and the design makes the tradeoff between latency, throughput, area, and timing easy to measure. 

## Design Summary

The final RTL design is a 16-stage fully pipelined CORDIC accelerator.

- Input: fixed-point angle in radians
- Outputs: 16-bit Q1.15 cosine and sine
- Internal datapath: 20-bit signed fixed point with 16 fractional bits
- Latency: 16 cycles
- Throughput: one sine/cosine output pair per clock after the pipeline is full

The tested input angle range is from approximately `-pi/2` to `+pi/2`.

## RTL Simulation and Quartus Compile

The RTL was simulated in ModelSim using `tb/tb_cordic_pipelined.sv`. The testbench reads `tb/test_vectors.txt`, drives each input angle into the RTL, waits for `valid_out`, and compares `cos_out` and `sin_out` against the Python-generated expected values.

The ModelSim run loaded 11 test vectors and completed with all tests passing. The simulation log is included in:

`results/modelsim_rtl_sim.log`

The RTL was also compiled in Quartus Prime Lite Edition for a Cyclone V FPGA target. Quartus full compilation completed with 0 errors. With the 10 ns SDC clock constraint in `cordic_pipelined.sdc`, Quartus reported positive setup slack at all timing corners.

The Quartus compile summary is included in:

`results/quartus_compile_summary.md`

## Synthesis Result Summary

I synthesized the design on the RPI ECSE `ts3` server using Synopsys Design Compiler NXT through `dcnxt_shell`.

The fastest tested clock period that met timing was 10 ns, which corresponds to 100 MHz. The 10 ns run met timing with 0.00 ns slack. A slightly faster 9.5 ns clock period failed with -0.11 ns slack, so I treated 100 MHz as the maximum verified frequency for this version of the design.

At the 10 ns implementation point:

- Total cell area: 34969.000000
- Reported total power: 0.3320 mW
- Latency: 16 cycles
- Steady-state throughput: 100 million sine/cosine output pairs per second

The reported power should be treated as a synthesis-level switching-power estimate because the target library did not characterize internal power.

`results/synthesis_summary.md`
